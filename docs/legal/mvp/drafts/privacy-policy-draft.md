## 1. Назначение документа

Этот черновик описывает privacy boundary для MVP: сотрудник получает доступ к образовательному B2B-пилоту, а HR/wellbeing sponsor получает отчеты о запуске и вовлеченности без персонального контроля чувствительных учебно-финансовых ответов по умолчанию.

Документ не заменяет финальную политику конфиденциальности, договорную документацию с заказчиком, уведомление/реестр оператора персональных данных или отдельные согласия.

## 3. Какие данные предполагаются в MVP

MVP может использовать следующие категории данных только в пределах подтвержденного пилотного сценария:

- идентификация и доступ: имя, email, телефон, индивидуальный invite code, связь с tenant / pilot launch / access pool;
- учебные события: старт и завершение уроков, мини-тесты, прогресс, выбранные практические шаги;
- диагностика: ответы, уровень, weak zones и route profile для построения образовательного маршрута;
- points: начисления, списания, баланс и корректировки как игровая система прогресса, не денежный баланс;
- merch: заказ, статус, выбранная позиция и контактные данные для выдачи физической награды;
- support/feedback: оценки, обращения, категории вопросов и статус обработки.

MVP не должен требовать точные суммы дохода, долгов, расходов, фото документов, банковские скриншоты или персональные финансовые отчеты для прохождения обучения.

## 4. Цели использования данных

Данные используются для:

- регистрации и восстановления доступа в рамках invite-code сценария;
- построения образовательного маршрута и отображения прогресса сотруднику;
- работы points, магазина merch и операционной выдачи заказов;
- поддержки пользователя и обработки feedback;
- агрегированной аналитики пилота для HR/wellbeing sponsor;
- безопасности, аудита admin-доступа, антифрода и технической поддержки.

Новые цели обработки, особенно связанные с персональными HR-отчетами или customer-specific analytics, требуют отдельного human review.

## 5. Граница видимости для HR / sponsor

Без отдельного согласованного сценария HR/wellbeing sponsor получает агрегированные отчеты по пилоту:

- приглашения, регистрации, активацию и retention;
- completion и learning progress в агрегированном виде;
- diagnostic insights в агрегированном виде;
- engagement, rewards, support categories и pilot outcome summary;
- launch/access-pool разрезы только при соблюдении privacy threshold.

По умолчанию персональные diagnostic answers, индивидуальные weak zones, exact sums, reflection details, личные налоговые/долговые обстоятельства и содержимое support-обращений не являются персональными HR-отчетами.

Персональные merch-данные могут быть доступны операционным ролям только настолько, насколько это нужно для выдачи физической награды и антифрода.

## 6. Доступ администраторов

Product/admin доступ к чувствительным данным должен быть ролевым, ограниченным целью поддержки/аналитики/настройки продукта и журналироваться. Production admin policy, роли, сроки хранения audit logs и customer-specific доступы остаются human-gated.

## 7. Хранение, локализация и удаление

Финальная политика должна определить:

- сроки хранения по категориям данных;
- порядок удаления или обезличивания;
- правила резервного копирования;
- требования к хранению и первичной обработке персональных данных граждан РФ с использованием баз данных на территории РФ;
- необходимость уведомления Роскомнадзора до запуска обработки.

Этот черновик фиксирует продуктовую необходимость проверки, но не утверждает выполнение этих требований.
